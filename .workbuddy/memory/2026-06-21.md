## 任务：诊断所有优化器效果低于 baseline 的根因

### 背景
benchmark 显示所有 12 个优化器变体（ape/opro/dspy/promptbreeder/evoprompt/spo 各 2 个别名）的 accuracy 均低于 baseline 0.85，最好的 spo-optimizer 只有 0.815（-3.5pp）。部分方法耗时极高（SPO 24000+ 秒）。

### 诊断结论

baseline 和优化器走两条不同的评估链路，优化器链路存在 3 个关键 bug + 2 个设计缺陷：

**P0 致命 Bug（直接解释 6pp 差距）：**
1. **`{input}` 占位符未替换** — `src/prompt_evolution/evaluation/evaluator.py:123`
   - 初始 prompt 模板 `新闻标题："{input}"\n类别：` 需要 `replace("{input}", title)`
   - baseline (`benchmark_news.py:265`) 正确做了替换 → 0.85
   - 优化器走 `Evaluator._generate_predictions`，直接 `f"{instruction}\n\n用户输入：{user_input}"`，`{input}` 字面量原样发给 LLM

2. **instruction 双发** — `evaluator.py:125-130`
   - `system_prompt=prompt.instruction` + `prompt=full_prompt`（含 instruction），LLM 收到 instruction 两次

3. **输出格式引导被破坏** — 初始 prompt 末尾 `\n类别：` 被追加的 `\n\n用户输入：xxx` 打乱，配合 AccuracyMetric 严格 `==` 比较导致误判

**P0 设计缺陷：**
4. 候选 prompt 生成时未约束保留 `{input}` 占位符 — APE/OPRO/SPO 改写 prompt 时可能丢掉占位符

**P1：**
5. Evaluator 无并发，顺序 `for item in dataset` — SPO 14400 次串行调用 → 24000 秒

**P2：**
6. 训练集仅 200 条，评估方差大
7. 初始 prompt 已强（0.85），优化空间有限

### 修复优先级
- P0：修 Evaluator 三处 bug + 候选生成约束 → 优化器准确率应追平 baseline
- P1：Evaluator 加并发 → SPO 耗时 24000s → ~2000s
- P2：扩训练集 / 换更难数据集

### 关键文件
- `src/prompt_evolution/evaluation/evaluator.py` — 评估器核心（P0 主修目标）

## P0 修复实施

### P0-1: PromptBreeder / EvoPrompt 候选评估失效

**根因**：`if candidate.score is None:` 恒为 False（`PromptCandidate.score` 默认 0.0，是 float 非 Optional），导致变异/交叉子代从未被评估。

**修复方案**：给 `PromptCandidate` 加 `evaluated: bool = False` 标记字段，在 `Evaluator.evaluate` 内部评估后自动设置 `prompt.evaluated = True`，进化算法改用 `if not candidate.evaluated:` 判断。

**改动文件**：
1. `src/prompt_evolution/core/models.py` — PromptCandidate 加 `evaluated: bool = False` 字段
2. `src/prompt_evolution/evaluation/evaluator.py` — `evaluate()` 返回前设置 `prompt.score = final_score; prompt.evaluated = True`（原地回写，所有优化器自动受益）
3. `src/prompt_evolution/optimizers/prompt_breeder/optimizer.py`
   - `unevaluated = [p for p in population if p.score is None]` → `if not p.evaluated`
   - 删除冗余的第二个评估循环（dead code）
   - 主循环结束后加"最终评估"：对 all_candidates 里所有未评估候选（最后一轮子代）做一次评估，确保 best_prompt 选择公平、num_candidates_evaluated 准确
   - `num_candidates_evaluated` 统计改用 `c.evaluated`
4. `src/prompt_evolution/optimizers/evoprompt/optimizer.py` — 同上三处改动

**验证**：
- 9 个现有测试全过
- `scripts/verify_p0_fix.py` 用 mock provider 验证：修复前子代 score 全 0.0、best 选 initial；修复后所有候选 evaluated=True，best_prompt score=1.0 且选到含 MAGIC 关键词的高分子代

### P0-2: benchmark 训练集/测试集口径混比

**根因**：baseline 报测试集准确率，优化器报 `result.best_prompt.score`（训练集得分），二者并列对比导致"优化器都不如 baseline"结论失真。

**修复**：`benchmark_news.py` 优化器循环重写：
1. 每个优化器跑完后，用 `best_prompt.instruction` 在 `test_data` 上调用 `evaluate_prompt` 评测（与 baseline 同口径），用独立的 `method_name=f"{method}__test"` 做 checkpoint key
2. `results` 里 `score` = 测试集准确率（主指标），新增 `train_score` 字段（参考指标）
3. checkpoint 改用 `test_accuracy` 字段；跳过逻辑只在 `test_accuracy` 存在时才完全跳过（旧 checkpoint 只有 `accuracy`=训练集分数，会被重跑——这是正确的，旧数据口径错了）
4. 汇总表显示 `Test Acc` + `Train Acc` 两列，并加注释说明口径

**验证**：语法检查 + 导入检查 + 关键改动点断言全部通过

- P0-1 是前几轮诊断漏掉的 bug：之前只修了 evaluator 链路（占位符/双发/并发）和候选生成的 `{input}` 约束，但没发现进化算法的评估判断条件本身是 dead code

### 关键洞察
- F1ScoreMetric 的 set→Counter 是经典坑：set 法对"重复词多的预测"会系统性高估 precision
- OPRO `[-20:]` 取最低分这个 bug 直接影响 OPRO 效果——LLM 看到的是最差的 20 条历史 prompt，难以学到好的优化方向
- litellm.drop_params=True 模块级副作用在多库共用 litellm 的场景会引发意外行为，改为 per-call 是更安全的做法
- dspy_optimizer 自实现不依赖 dspy 库，但 pyproject 还挂着 dspy-ai>=2.5.0 依赖（重依赖），删掉能显著瘦身安装

### 关键洞察
- 测试设计模式：用 mock provider 让"含 MAGIC 关键词的 prompt 预测正确，否则错误"，从而让"初始 prompt 0 分 / 子代满分"成为可验证的不变量——这恰好能捕获 P0-1 类 bug（子代未被评估 → best 仍是 0 分初始）
